# 手冲手记 · 云同步设计

> 日期：2026-06-05
> 目标：让手机和电脑自动同步咖啡数据，告别每次手动导出/导入。

## 1. 背景与现状

当前软件是纯本地 PWA，数据存在浏览器 `localStorage`（键名 `coffee-log-v1`），结构为
`{ version, beans:[], brews:[] }`。每条记录有 `id` 和 `createdAt`，但**没有最后修改时间**，删除是直接从数组里抹掉。跨设备只能靠「导出 JSON → 另一台导入」。

用户希望：手机和电脑**两边都经常记**，数据自动合到一起、一条不丢、不用手动操作。

## 2. 总体架构

```
┌─────────────┐         ┌──────────────────────┐         ┌─────────────┐
│   手机浏览器  │ ←─────→ │   Cloudflare 云端      │ ←─────→ │  电脑浏览器   │
│  (PWA 打开)  │  HTTPS  │  Pages: 放网页         │  HTTPS  │  (PWA 打开)  │
│ localStorage │         │  Worker: 同步 API      │         │ localStorage │
│ 本地仍是主存  │         │  KV: 存数据            │         │ 本地仍是主存  │
└─────────────┘         └──────────────────────┘         └─────────────┘
```

- **Cloudflare Pages**：托管静态网页文件（`index.html` / `app.js` / `style.css` 等），手机电脑用同一个网址打开。
- **Cloudflare Worker（Pages Function）**：提供同步 API（拉取 / 推送）。
- **Cloudflare KV**：键值存储，按「同步码」分格存放每个用户的数据。
- **本地 localStorage 仍是主存**：离线照常记录，联网后自动同步。云端是「中转 + 备份」。

选 Cloudflare 的理由：免费、不需信用卡、国内可访问、Pages + Worker + KV 一个账号搞定。

## 3. 数据模型变更

为支持「逐条合并」，给每条记录（beans / brews）增加字段：

| 字段 | 含义 | 何时写入 |
|---|---|---|
| `updatedAt` | 最后修改时间（毫秒时间戳） | 创建时 = `createdAt`；每次编辑时更新为 `Date.now()` |
| `deleted` | 软删除标记（`true`/不存在） | 删除记录时设为 `true` 并更新 `updatedAt`，**不再从数组移除** |

删除豆子时连带删除其冲煮记录的逻辑保留，但同样改为软删除（给每条相关 brew 打 `deleted` 标记）。

渲染层需过滤掉 `deleted` 的记录（用户看不到墓碑）。

向后兼容：旧数据没有 `updatedAt` 的记录，加载时补 `updatedAt = createdAt || 0`，`deleted` 视为 `false`。

## 4. 合并规则（方案 B：逐条合并 / 末次修改胜出）

合并两份数据（本地 + 云端）时，对 beans 和 brews 各自按 `id` 配对：

1. **只有一边有** → 保留该条。
2. **两边都有同一 `id`** → 比较 `updatedAt`，**新的胜出**（含 `deleted` 状态一并以新的为准）。
3. **`updatedAt` 相等** → 任取其一（数据相同，无所谓）。

墓碑（`deleted:true` 的记录）**永久保留在数据中**，确保删除能传播到所有设备且不会死灰复活。对个人用量，墓碑数量可忽略，不做清理（YAGNI）。

合并是**纯函数**：输入两份数据，输出合并结果，不依赖外部状态，便于单测。

## 5. 同步流程

### 客户端（浏览器）

- **打开软件时**：`pull`（拉云端）→ 与本地 `merge` → 写回 localStorage → `push`（推合并结果到云端）。结果：一开就是最新。
- **每次存档后**（新增/编辑/删除）：本地先存（保证离线可用），再尝试 `push`。
- **网络恢复时**（`online` 事件）：自动补跑一次「pull → merge → push」。
- **离线时**：只写本地，标记「有待同步改动」；联网后自动补同步。

为避免两台设备几乎同时 push 互相覆盖：push 前先 pull+merge，推送的是合并后的全量数据。单人场景同时写极罕见，不做乐观锁（YAGNI），但 Worker 端 push 时也会与 KV 现有数据再合并一次兜底（见下）。

### 服务端（Worker API）

- `GET /api/data`：校验同步码 → 返回 KV 中该用户的数据（没有则返回空 `{beans:[],brews:[]}`）。
- `POST /api/data`：校验同步码 → 将请求体与 KV 现有数据**再合并一次**（用同一套合并规则）→ 写回 KV → 返回合并结果。
  - 服务端二次合并是兜底：即便两台设备并发 push，也不会互相覆盖丢数据。

## 6. 鉴权（同步码）

- 用户自设一串「同步码」（如 `mycoffee-8829`），手机和电脑各填一次相同的码。
- 同步码同时承担两个职责：
  - **身份口令**：每个请求带上同步码（HTTP 头，如 `X-Sync-Code`）；Worker 校验，缺失或为空一律拒绝（401）。
  - **数据归属**：KV 的存储键由同步码派生（如对同步码做哈希得到 key），不同同步码互不可见。
- 同步码存在本地 localStorage（单独键，如 `coffee-sync-code`），不随业务数据导出。
- 安全边界：这是「弱口令隔离」，足够防止陌生人随便翻看，不是高强度加密。用户需自选一串不易猜的码。丢了码 = 换新格子，旧数据仍在云端，重填旧码即可找回。

## 7. 界面变更

- 在「设置 / 备份」区域新增「云同步」小面板：
  - 一个输入框填**同步码** + 「保存并同步」按钮。
  - 一行**同步状态**：如「✅ 已同步（刚刚）」「🔄 同步中…」「📴 离线，待同步」「⚠️ 同步码错误」。
- 原有「导出备份 / 导入」按钮**保留**，作为离线备份后路。

## 8. 错误处理

| 情况 | 处理 |
|---|---|
| 离线 / 网络失败 | 静默：只写本地，状态显示「离线，待同步」，联网后自动补 |
| 同步码为空 | 不发起同步，仅本地模式（等同现在的纯本地） |
| 同步码校验失败（401） | 状态显示「同步码错误」，不覆盖本地数据 |
| 云端返回数据损坏 | 跳过合并，保留本地，控制台告警 |
| 本地 localStorage 写入失败 | 沿用现有兜底逻辑 |

核心原则：**任何同步异常都不能弄坏或丢失本地数据**；本地永远是安全的主存。

## 9. 部署方式

- 用户的环境已有 Node 22，可用 `npx wrangler` 部署，无需 GitHub。
- 步骤（实现阶段提供逐步脚本）：
  1. 注册 Cloudflare 免费账号；
  2. `wrangler login`；
  3. 创建 KV 命名空间，写入 `wrangler.toml` 绑定；
  4. `wrangler pages deploy`（含 Pages Function）一次部署，拿到网址；
  5. 手机电脑打开网址，各填一次同步码。

## 10. 不做的事（YAGNI）

- 不做多用户账号体系 / 注册登录（单人，同步码足够）。
- 不做完整改动历史 / CRDT 字段级合并。
- 不做墓碑清理 / 垃圾回收。
- 不做端到端加密（弱口令隔离够用，可作为未来增强项）。

## 11. 测试要点

- 合并纯函数单测：新增/编辑/删除三类、时间先后、墓碑传播、空数据、缺字段旧数据兼容。
- 手动验收：A 设备记 3 杯 + B 设备记 2 杯离线，各自联网后两边都应看到 5 杯；A 删一杯，B 同步后该杯消失且不复活。
