# 内置冲煮计时器 · 设计文档

**日期**: 2026-06-05  
**状态**: 待评审

## 概述

在已有「记录一杯」表单的基础上，增加一个**全屏分段计时器**。用户在冲煮前先填好豆子/粉量/水量等参数，然后进入全屏计时模式；冲完后自动回到表单，总时长和闷蒸时间已自动填入。

核心设计原则：**冲的时候手占着，界面上每一步都只需要一个动作。**

---

## 1. 架构与文件分工

| 文件 | 改动 | 职责 |
|------|------|------|
| `timer.js` | 🆕 新增 | 计时器核心——状态机、墙上时钟计时、全屏界面控制。不碰数据存储。 |
| `index.html` | 改 | 全屏计时界面的骨架（默认隐藏）；「记录一杯」弹窗加「▶ 开始冲煮计时」按钮；引入 `timer.js` |
| `app.js` | 改 | 两段胶水：① 点按钮 → 启动计时器；② 计时器回调 → 时间填回表单 |
| `style.css` | 改 | 全屏界面的样式 |

### 接口（timer.js ↔ app.js）

```
启动：BrewTimer.start({ beanName, onFinish })
回传：onFinish({ totalSec, bloomSec, stages, aborted })
```

- `totalSec`：总冲煮时间（累计秒数）
- `bloomSec`：闷蒸结束时的累计秒数（null = 没点闷蒸）
- `stages`：`[{ label, sec }]`——每段结束时刻（累计秒）。最后一段的就是 `totalSec`
- `aborted`：用户中途退出，为 true 时不填表单

`timer.js` 不知道「豆子」「评分」是什么，只知道「测时间、报节点」；`app.js` 不知道计时器内部怎么数秒。

### 回调 → 表单字段映射

计时器抓到的数据**直接复用现有字段**，不加新字段类型：

| 计时器产出 | 存进什么字段 | 说明 |
|-----------|-------------|------|
| `totalSec` | `brew.totalTime` | 秒→"分:秒"字符串，复用已有字段和 `formatTime()` |
| `bloomSec` | `brew.bloomTime` | 已有字段，秒数 |
| `stages` | `brew.stages`（新字段） | 数组，老记录无此字段时兜底空数组 |

因为「找规律」分析本来就读 `bloomTime` 和 `totalTime`（见 `app.js` 的 `NUMERIC_METRICS`），**计时器上线后分析代码一行不用改。**

## 2. 全屏界面设计

专为「一手拿壶、瞄一眼手机」设计：超大数字、大按钮、只显示必要信息。

### 状态① 准备（进来还没点开始）

只显示计时 0:00 和大「开始」按钮。顶部有「← 退出」，点完直接退（没开始计时，无需确认）。

### 状态② 闷蒸中（点了开始）

- 顶部：豆子名 + 「← 退出」
- 中间：超大总时间数字（实时走），下方「闷蒸中…」
- 分段列表：`① 闷蒸  进行中…`
- 底部两个按钮：「闷蒸结束」+「停止」

### 状态③ 注水中（点过闷蒸结束）

- 中间：超大总时间数字，下方「注水中…」
- 分段列表：`① 闷蒸  0:32 ✓`（已定格）+ `② 注水  进行中…`
- 底部两个按钮：「下一段」+「停止」
- 继续点「下一段」→ 生成 ③、④… 自由分段

### 界面要点

- 主操作按钮（左边的）是**同一个按钮**，文字随阶段变化：准备态无 → 闷蒸态「闷蒸结束」→ 注水态「下一段」。同一时刻只有一个「往前走」的选择。
- 分段列表用「✓ 已定格 / 进行中…」区分，给用户确定感。
- 「停止」→ 界面淡出 → 自动回到「记录一杯」表单。
- 不想细分就不点什么「下一段」，直接「停止」即可。

## 3. 完整用户流程

```
「冲煮记录」页 → 点「+ 记录一杯」
  → 弹窗打开：选豆子、填粉量/水量/研磨/水温（开冲前准备）
  → 点「▶ 开始冲煮计时」
  → 弹窗暂时隐藏，全屏计时界面淡入
  → 闷蒸、注水、分段、停止（计时过程）
  → 全屏淡出，弹窗重新出现
  → 总时间/闷蒸时间已自动填好（编辑框闪烁提示）
  → 补评分、笔记、味道 → 保存
```

`stages` 分段数据在计时器停止到用户保存之间，先临时存在内存变量里，保存时一并写入记录。

## 4. 数据与展示

### 新增字段

```js
brew.stages = [
  { label: "闷蒸", sec: 32 },
  { label: "段2",  sec: 80 },
  { label: "段3",  sec: 150 }
]
```

每项 `sec` 是**累计秒**（从开始计时起算），不是本段耗时。这样既能算每段耗时（相邻相减），也能知道各段在冲煮中什么时刻发生。

老记录没有 `stages` 字段 → 兜底空数组 `[]`，与 `tasteIssues` 处理方式一致。

### 展示位置

- **冲煮记录卡片**：加一行「节奏」—— `闷蒸 0:32 · 段2 0:48 · 段3 1:10`（每段耗时，自动算）
- **单豆详情「调试轨迹」**：这条记录可看到注水节奏，与评分对照

## 5. 异常兜底

| 情况 | 处理 |
|------|------|
| 手机息屏 | 用 Wake Lock API 申请屏幕常亮；不支持时降级处理，计时不受影响 |
| 切到后台后再回来 | 用「墙上时钟」算时间（记开始的真实时刻），不用 `setInterval` 累加。回来时间依然准 |
| 点「← 退出」 | 已开始计时则弹确认「放弃这次计时？已记的时间会丢」；未开始直接退 |
| 误刷新/误关页面 | `beforeunload` 拦一道：「计时进行中，确定离开？」 |
| 没点「闷蒸结束」就停止 | 闷蒸留空，总时间照填，分段记成一整段。不报错 |
| 手抖连点「下一段」 | 1 秒内的第二次点击忽略，防止冒出一个 0 秒段 |
| 忘了停、冲到几十分钟 | 照常计，`formatTime` 支持到 99:59 |
| 刷新后恢复进行中计时 | **v1 不做**（YAGNI）。恢复弹窗状态 + 计时状态复杂度高，留给以后 |

### 明确不做的

- 刷新页面后**恢复进行中的计时**——需要同时恢复弹窗里的豆子/粉水等，边界太多
- 计时过程**标注每段水量**——没有蓝牙秤无法边冲边输

## 6. 测试策略

### 单元测试（`timer.test.js`，Node 跑）

把核心逻辑抽成纯函数 `buildResult(timestamps)`：
- 正常流程：开始0 → 闷蒸结束32 → 停止150 → 闷蒸32s、总150s、stages 两段
- 没点闷蒸：开始0 → 停止180 → 闷蒸 null、总180s
- 只点一下就停：开始0 → 停止45 → 闷蒸=总时间,一段
- 连点防抖：1秒内多次点同一段 → 只记第一次
- 空输入边界

### 浏览器实测

- 完整流程走一遍（准备 → 闷蒸 → 分段 → 停止 → 表单回填）
- Wake Lock 申请确认
- beforeunload 拦截确认
- 控制台零错误零警告

## 7. 对现有功能的兼容性

- **分析页**：读 `bloomTime`/`totalTime` 的逻辑不变，计时器产出的数据格式与之完全兼容
- **云同步**：`stages` 是新字段，老数据无此字段 → 合并/导出/导入都兜底空数组
- **PWA 离线**：计时器是纯前端功能，不依赖网络
- **导出/导入**：新字段 `stages` 自动包含在 JSON 中，无需特殊处理
